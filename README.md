# Procedural Music Reasoning

Procedural generators for music-theory reasoning tasks aimed at language-model
benchmarking, supervised fine-tuning, and reasoning-trace training.

The project is designed as a companion package to
[Reasoning Core](https://github.com/sileod/reasoning-core). It keeps the same
core task interface:

```python
from music_reasoning_tasks import get_task

task = get_task("pitch_interval_reasoning")
example = task.generate_example(level=5)
score = task.score_answer("minor third", example)
```

Each generated example contains:

- a natural-language prompt,
- a compact expected answer,
- metadata describing the generated instance, and
- a chain-of-thought style reasoning trace (`cot`) generated by the task
  generator.

## Motivation

Music theory offers a useful testbed for language-model reasoning because many
answers cannot be recovered from surface patterns alone. A model may need to
track pitch spelling, interval size, octave/register, key-signature context,
Roman-numeral function, chord inversion, and notation-specific conventions.
Small changes in notation can change the correct answer, while enharmonically
similar objects may still require different written responses.

Hand-written benchmark sets are difficult to scale across all of these cases.
This package uses procedural generation to create large, controlled datasets
with known answers and generated reasoning traces. It is intended for
benchmarking, supervised fine-tuning, and studying whether language models can
follow explicit music-theory rules.

The generators therefore emphasize:

- precise symbolic answer formats,
- diverse prompt formulations,
- multiple notation systems,
- controlled procedural difficulty,
- explicit reasoning traces, and
- deterministic scoring.

The current implementation includes two task families:
`pitch_interval_reasoning` and `chord_roman_reasoning`. See
`docs/music_reasoning_specification.md` for their modes and difficulty knobs.

## Notation Coverage

The generators use several notation styles:

- **Scientific pitch notation (SPN)**, for example, `C#4` or `Bbb2`.
- **Compact ABC note tokens**, for example, `^C`, `_B,,`, or `=F'`.
- **Full ABC score fragments**, for example:

  ```abc
  L:1/8
  M:2/4
  K:Eb
    [^A=BD^F] |] %1
  ```

Full ABC score fragments are interpreted using their `K:` key signature. When
key signatures affect note meaning, the generated reasoning trace resolves the
implicit accidental explicitly.

## Difficulty Levels

The `level` argument controls the distribution of generated data. Higher levels
increase the probability or range of harder features, such as:

- more accidentals,
- more complex keys,
- longer interval/transposition chains,
- larger interval numbers,
- more ABC notation,
- more secondary/applied Roman numerals.

This is slightly different from many Reasoning Core task families where a
single example at level `N` is usually a task of level `N`. Here, a dataset
generated at level `N` is a richer distribution that may still include easier
instances. That is intentional: music curricula naturally mix easier and harder
subskills while gradually increasing the frequency of difficult cases.

## Installation

From a checked-out repository:

```bash
pip install -e ".[test]"
```

The package depends on:

- `reasoning_core` for the task/config/problem interface,
- `music21` for validation and selected chord/Roman-numeral operations.

If Reasoning Core is not available from your package index, install it directly
from its GitHub repository before installing this project.

## Usage

Generate one example:

```python
from music_reasoning_tasks import get_task

task = get_task("chord_roman_reasoning")
example = task.generate_example(level=5)

print(example.prompt)
print(example.answer)
print(example.metadata.cot)
```

Score an answer:

```python
score = task.score_answer("V65/V", example)
```

List available task families:

```python
from music_reasoning_tasks import list_tasks

print(list_tasks())
```

## Generate Datasets

Use the included script:

```bash
python scripts/generate_examples.py pitch_interval_reasoning \
  --level 5 \
  --num-examples 5000 \
  --out-dir generated_data \
  --readable

python scripts/generate_examples.py chord_roman_reasoning \
  --level 5 \
  --num-examples 5000 \
  --out-dir generated_data \
  --readable
```

This writes JSONL files suitable for training/evaluation and optional readable
text files for manual inspection.

## Generate A Balanced Benchmark

To generate the proposed easy/moderate/hard benchmark with 200 examples per
mode, run:

```bash
python scripts/generate_benchmark.py \
  --levels 0 3 5 \
  --examples-per-mode 200 \
  --out-dir benchmark_data/music_reasoning_benchmark \
  --seed 0
```

The script writes Hugging Face-friendly split files under `data/`, a
`summary.json`, and a dataset-card draft `README.md`.

Benchmark exports append `Return only the requested answer.` to every prompt
to enforce compact evaluation responses. This instruction is benchmark-only;
examples generated directly for training retain their original prompts.

## Evaluate an OpenRouter Model

Install the benchmark-evaluation dependency:

```bash
pip install -e ".[benchmark]"
```

Paste an OpenRouter API key into `OPENROUTER_API_KEY` near the top of
`scripts/evaluate_openrouter.py`, then provide an exact OpenRouter model ID:

```bash
python scripts/evaluate_openrouter.py openai/gpt-4.1-mini
```

By default, the script evaluates every row in the `easy`, `moderate`, and
`hard` splits downloaded from
[`dpechenev/music-reasoning-benchmark`](https://huggingface.co/datasets/dpechenev/music-reasoning-benchmark),
using its latest stable semantic-version tag. The selected tag and exact commit
are recorded in every result and report. Calls are made in bounded batches through
[`litlm`](https://github.com/sileod/litlm), and each response is evaluated with
the corresponding task family's `score_answer` method. All predictions are
stored in one resumable JSONL file and one detailed Markdown report for
convenience, while every metric remains separated by difficulty split. Outputs
are organized as:

```text
benchmark_results/<model>/<version>/
  results.jsonl
  report.md
```

The report includes recorded API benchmark time separately for each split.

Select another published benchmark release with `--revision`, for example:

```bash
python scripts/evaluate_openrouter.py openai/gpt-4.1-mini \
  --revision v0.1.1
```

For models that expose configurable reasoning through OpenRouter, set an effort
level explicitly:

```bash
python scripts/evaluate_openrouter.py deepseek/deepseek-v4-pro \
  --reasoning-effort high \
  --max-tokens 4096
```

Effort-specific evaluations are stored in a subdirectory such as
`reasoning-high/` so they remain separate from default-effort runs.

Before launching a full run, an inexpensive smoke test is recommended:

```bash
python scripts/evaluate_openrouter.py openai/gpt-4.1-mini \
  --limit-per-split 5
```

## Testing

Run the full test suite:

```bash
pytest
```

Run with coverage:

```bash
pytest --cov=music_reasoning_tasks --cov-report=term-missing
```

The tests check:

- generated examples for every mode,
- score normalization,
- ABC parsing behavior,
- interval and note-spelling logic,
- chord/Roman-numeral behavior against `music21`,
- prompt/answer consistency.

## Documentation

The `docs/` directory contains two higher-level project documents:

- `music_reasoning_specification.md` describes the implemented task families,
  modes, difficulty knobs, metadata conventions, and planned extensions.
- `music_reasoning_overview_report.md` is an overview of the
  current implementation, including task-family summaries, diversity estimates, and
  testing/coverage information.

## Extending the Project

To add a new music task family:

1. Add a new module under `src/music_reasoning_tasks/`.
2. Subclass `reasoning_core.template.Task`.
3. Reuse shared notation/theory helpers from `_music_theory.py`.
4. Add the task class to `TASK_CLASSES` in `music_reasoning_tasks.__init__`.
5. Add focused tests under `tests/`.
6. Update `docs/music_reasoning_specification.md`.
7. Update `scripts/generate_benchmark.py` so the new family can be included in
   balanced benchmark generation.

Prefer putting reusable music-theory primitives in `_music_theory.py`; keep
family-specific sampling and prompt logic in the task-family module.
